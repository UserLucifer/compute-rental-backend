本手册把当前后端系统拆成产品经理能直接校准的业务流程：每个业务从哪里开始、经过哪些状态、由谁触发状态变化、何时闭环、账务如何发生、哪些地方当前代码已有实现、哪些地方需要上线前确认。

1. 从 0 认识系统

这是一个“用户用钱包余额购买算力机器租赁权益，再通过 AI 模型收益规则获得每日收益，到期或提前结束后返还本金并结算收益”的平台。系统还支持邀请关系、团队收益佣金、充值、提现、后台审核、内容文章、文档、站内消息等配套业务。

用户在前台做什么

  • 注册、登录、重置密码。
  • 浏览地区、GPU、机器商品、AI 模型、租赁周期。
  • 充值 USDT，等后台审核入账。
  • 创建租赁订单，支付机器租赁费。
  • 支付 API 部署费后订单立即运行，24 小时后系统自动暂停，用户可再次启动。
  • 订单运行后每天获得收益，到期或提前结算后返还本金。
  • 查看收益、佣金、团队、钱包流水，申请提现。

管理员在后台做什么

  • 维护商品、地区、GPU 型号、AI 模型、租赁周期。
  • 维护充值渠道，审核充值订单。
  • 审核提现、确认打款。
  • 查看用户、订单、钱包流水、收益、佣金、团队和看板。
  • 维护系统参数，手动触发定时任务。
  • 维护博客、文档、站内消息。

系统自动做什么

  • 根据 JWT 识别用户/管理员身份。
  • 控制邮箱验证码频率与有效期。
  • 用钱包流水记录每一笔资金变化。
  • 订单支付后生成 API 凭证，并在超时未部署时取消。
  • API 部署付款后 24 小时自动暂停，等用户启动。
  • 每日定时结算租赁收益与上级佣金。
  • 订单到期后自动返还本金并关闭订单。
阅读方式：产品校准时建议先看第 4 节主业务闭环，再逐个看第 8 至第 13 节资金和订单流程，最后用第 18 节清单逐条确认需求是否一致。

2. 业务地图与角色

2.1 角色边界

角色 可做的事情 业务边界
游客 查看公开商品、地区、GPU、AI 模型、租赁周期、博客、文档；提交租赁收益测算。 不能创建订单、不能使用钱包、不能查看个人收益。
注册用户 充值、创建租赁订单、支付机器费、支付部署费、启动订单、提前结算、提现、查看团队和佣金。 需要账号状态为启用；部分服务还会校验钱包启用。
管理员 管理后台数据、审核充值提现、查看经营数据、触发定时任务、维护内容和站内消息。 后台接口需要管理员角色；管理员退出当前只记录日志，不做 token 黑名单。
系统定时任务 超时取消、自动暂停、每日收益、到期结算、佣金生成。 依赖数据库状态与系统时间；失败会影响业务推进，需要运营监控。

层级 包含模块 产品理解

3. 全局规则

3.1 登录与权限

接口类型 权限规则 备注
公开接口 /api/auth/**、博客、文档、健康检查、公开商品查询、租赁测算等不要求登录。 游客可访问公开商品和内容。
用户接口 除公开和后台接口外，默认需要用户 JWT，角色为 ROLE_USER。 用户登录后拿到 Bearer Token。
后台接口 /api/admin/** 需要管理员 JWT，角色为 ROLE_ADMIN。 管理员登录与退出会写后台日志。
已闭环：JWT 过滤器每次请求都会重新查询用户/管理员当前状态。用户或管理员被禁用后，旧 token 不再建立认证，受保护接口会按未登录处理。当前仍未实现主动退出 token 黑名单，退出主要依赖前端丢弃 token 和过期时间。

3.2 时间、金额与配置

规则 当前实现 业务影响
业务时区 统一使用 Asia/Shanghai。 每日收益、提现日限额、到期结算都按北京时间理解。
金额精度 钱包金额使用 8 位小数，舍入方式为 HALF_DOWN。 USDT、收益、佣金、罚金都按 8 位处理。
系统配置缓存 系统参数读取后缓存 10 分钟，后台更新会清缓存。 提现费率、最低充值等配置可运营调整。
默认充值最低金额 recharge.min_amount = 500。 若充值渠道也设置最低金额，取两者较大值。
默认提现最低金额 withdraw.min_amount = 10。 用户提现申请金额必须大于等于该值。
默认提现免手续费门槛 withdraw.fee_free_threshold = 100。 申请金额达到 100 时当前实现手续费为 0。
默认提现手续费率 withdraw.fee_rate = 0.05。 未达到免手续费门槛时，手续费 = 申请金额 * 5%。
默认每日提现上限 withdraw.max_daily_amount = 100000。 统计当天待审、已通过、已打款的提现申请金额。
默认待部署超时 order.pending_activation_timeout_minutes = 60。 机器费已付但部署费未付，超过 60 分钟自动取消并退机器费。

3.3 并发与幂等

  • 钱包更新使用版本号乐观锁，防止同一钱包并发扣款导致余额错乱。
  • 充值创建、提现创建、租赁支付、部署费支付等关键动作使用 Redis 锁，避免重复提交。
  • 钱包流水带唯一交易号和幂等键，重复业务动作应避免重复入账或扣款。
  • 收益记录有订单和收益日期唯一约束，佣金记录有来源收益和层级唯一约束。

3.4 多语言

商品、地区、GPU、AI 模型、租赁周期、充值渠道、博客、文档、通知等支持翻译表。业务接口通常按语言参数优先读取翻译，未命中时回退默认中文字段。

4. 主业务闭环

平台最核心的闭环是：用户注册获得钱包，充值后创建租赁订单，支付机器费，支付部署费，等待自动暂停后启动运行，系统每天发收益，上级每天拿佣金，到期自动返还本金；如果用户提前结束，则扣提前结算罚金后返还剩余本金。

4.1 主链路流程图

注册/登录用户通过邮箱验证码注册，系统创建用户、钱包、邀请关系。
→
充值入账用户提交充值凭证，后台审核通过后钱包可用余额增加。
→
选择商品选择地区、GPU、机器、AI 模型、租赁周期，系统计算预计收益。
→
支付机器费钱包扣机器租赁费，订单进入待激活并生成 API 凭证。
→
支付部署费钱包扣部署费，API 立即生效，订单进入运行中。
→
自动暂停部署付款 24 小时后系统自动暂停，等待用户启动。
→
用户启动订单进入运行中，收益周期开始计算。
→
每日收益/佣金每天定时给用户发租赁收益，并给上级发佣金。
→
结算闭环到期自动返还本金；提前结算则扣罚金后返还本金。

4.2 主链路泳道

用户
注册 → 充值 → 创建租赁订单 → 支付机器费 → 支付部署费并开始运行 → 24 小时后自动暂停 → 用户再次启动或提前结算 → 查看每日收益 → 到期收回本金 → 提现。
后台
配置商品/AI/周期 → 审核充值 → 审核提现 → 查看订单与财务数据 → 必要时手动触发定时任务 → 维护内容和消息。
系统
创建钱包和邀请链 → 扣款/入账写流水 → 生成 API 凭证 → 超时取消 → 自动暂停 → 每日收益 → 佣金生成 → 到期结算。

4.3 主链路状态总览

阶段 主要状态 触发人/触发器 资金变化 闭环结果
充值 SUBMITTED（已提交） → APPROVED（已审核通过） / REJECTED（已拒绝） / CANCELED（已取消） 用户提交、用户取消、后台审核 仅 APPROVED（已审核通过） 时钱包可用余额增加 用户获得可消费余额，或充值单终止
租赁订单 PENDING_PAY（待支付） → PENDING_ACTIVATION（待支付部署费） → RUNNING（运行中） → PAUSED（已暂停） → RUNNING（运行中） → EXPIRED（已过期） / EARLY_CLOSED（提前关闭） 用户支付、自动暂停任务、用户启动、到期任务、提前结算 支付机器费扣余额；支付部署费扣余额；结算返还本金 订单到期关闭或提前关闭
每日收益 PENDING（待处理） → SETTLED（已结算） 每日收益定时任务 钱包可用余额增加 收益入账后可提现或继续消费
佣金 PENDING（待处理） → SETTLED（已结算） 佣金定时任务 上级钱包可用余额增加 佣金入账后可提现或继续消费
提现 PENDING_REVIEW（待审核） → APPROVED（已审核通过） → PAID（已支付）；或 REJECTED（已拒绝） / CANCELED（已取消） 用户申请、用户取消、后台审核、后台打款 申请时冻结；取消/拒绝解冻；打款后扣冻结 链上打款完成或资金回到可用余额

5. 账号、邀请与团队

5.1 注册从哪里开始

注册从用户输入邮箱并发送注册验证码开始。系统先检查邮箱是否已经注册，未注册才允许发送验证码。验证码会写入数据库，并通过 Redis 控制短时间重复发送和每分钟发送次数。

发送注册验证码邮箱未注册，生成 SIGNUP（注册） 验证码，状态 UNUSED（未使用）。
→
校验验证码查找最新未使用、未过期验证码。
→
创建用户钱包用户状态 ENABLED（启用），钱包 USDT 可用/冻结均为 0。
→
创建邀请关系生成自己的邀请码，记录一级、二级上级。
→
返回 JWT用户注册后直接登录。

5.2 账号状态

状态 含义 如何产生 影响
ENABLED（启用） = 1 账号启用。 注册默认启用。 允许登录和正常操作；JWT 每次请求都会校验账号仍为 ENABLED（启用）。
DISABLED（禁用） = 0 账号禁用。 后台禁用用户。 不能重新登录；旧 JWT 立即失效；系统会暂停 RUNNING（运行中）订单、关闭运行片段并暂停 ACTIVE（生效中）API 凭证。
已确认口径：后台禁用用户时，会暂停 RUNNING（运行中）订单、关闭运行片段，并暂停 ACTIVE（生效中）API 凭证；不处理 PENDING_PAY（待支付）、PENDING_ACTIVATION（待支付部署费）、ACTIVATING（激活中）等状态。禁用后用户不会再被启用，旧 JWT 会被后端实时拦截。

5.3 邮箱验证码状态

状态 说明 触发方式
UNUSED（未使用） 验证码已生成，尚未使用。 发送注册验证码或重置密码验证码。
USED（已使用） 验证码已被消费。 用户注册成功或重置密码成功。
EXPIRED（已过期） 枚举存在，但当前主要通过过期时间判断，不主动把状态更新为 EXPIRED（已过期）。 当前代码未看到主动置为 EXPIRED（已过期） 的流程。

5.4 邀请和团队怎么形成

用户注册时如果填写邀请码，系统会找到直接邀请人，并把直接邀请人作为一级上级；直接邀请人的一级上级成为二级上级。当前代理体系最多只保存两级关系，第三层及更深层级不写入团队关系，也不参与佣金计算。

关系层级 来源 后续用途
一级上级 用户注册时填写的邀请码对应用户。 获得一级佣金。
二级上级 一级上级的一级上级。 获得二级佣金。
团队闭包关系 注册时最多写入 1、2 两层团队关系。 用于团队列表、团队人数统计；层级筛选只支持 1 和 2。
当前代理口径：代理链路已从原来的三级调整为二级。用户侧团队汇总只返回总团队数、直属团队数、二级团队数；后台用户团队查询也只统计两级。

6.3 收益测算公式

用户选择商品、AI 模型和周期后，系统会用当前配置计算预计收益。创建订单时这些配置会固化为订单快照，后续后台改商品价格或收益参数，不会自动改变已经创建的订单。

指标 公式 解释
预计每日收益 商品每日 token 产出 * AI 模型 token 单价 * 周期收益倍数 订单运行后每日收益也按这个快照公式生成。
预计总收益 预计每日收益 * 周期天数 仅是预计值，实际收益按每日定时任务逐天入账。
机器租赁费 商品租赁价格 支付机器费时从钱包可用余额扣除。
API 部署费 AI 模型部署费 支付部署费时从钱包可用余额扣除。
提前结算罚金 机器租赁费 * 周期提前结算罚金比例 提前结算时只影响返还本金，不从钱包额外扣余额。
已确认口径：商品库存、已租库存、可用库存和可租日期当前只作为后台展示/运营参考字段，不参与下单交易校验。创建订单只校验商品、AI 模型、周期是否启用。

7. 钱包与账务

7.1 钱包是什么

每个用户注册时会自动创建一个 USDT 钱包。钱包分为可用余额和冻结余额。可用余额能用于支付订单、支付部署费和申请提现；冻结余额主要用于提现审核期间锁住资金。

字段 业务含义 何时变化
availableBalance 可用余额。 充值、收益、佣金、结算返还会增加；支付机器费、部署费、提现冻结会减少。
frozenBalance 冻结余额。 用户申请提现时增加；取消/拒绝提现时减少；提现打款后扣减。
totalRecharge 累计充值。 充值审核通过入账时增加。
totalWithdraw 累计提现。 提现打款成功时按实际到账金额增加。
totalProfit 累计租赁收益。 每日收益入账时增加。
totalCommission 累计佣金收益。 佣金入账时增加。

7.2 钱包流水类型

流水类型 余额变化 典型业务
IN（入账） 可用余额增加。 充值入账、每日收益、佣金、订单结算返还本金、退款。
OUT（支出） 可用余额减少，或仅记录无余额变化的罚金流水。 机器费支付、部署费支付、提前结算罚金记录。
FREEZE（冻结） 可用余额减少，冻结余额增加。 用户提交提现申请。
UNFREEZE（解冻） 冻结余额减少，可用余额增加。 用户取消提现或后台拒绝提现。

7.3 业务类型

钱包流水还会记录业务类型，用来解释这笔钱为什么变化。当前已定义业务类型包括：充值、提现、租赁支付、API 部署费、租赁收益、佣金收益、结算返还、提前罚金、退款、后台调整。

校准点：ADJUST（调账） 后台调整枚举存在，但当前未看到后台人工调账接口。若上线需要人工调账，需要补齐审批、日志、权限和流水说明。

8. 充值业务

8.1 充值从哪里开始

充值从用户选择一个启用的充值渠道并提交充值订单开始。用户提交金额、链上交易号或凭证等信息后，充值单进入待审核。后台审核通过后，系统才给钱包入账。

用户选择渠道只能使用 ENABLED（启用） 的充值渠道。
→
提交充值单金额必须达到系统和渠道最低金额。
→
SUBMITTED（已提交）等待后台审核，用户可取消。
→
APPROVED（已审核通过）后台审核通过，按实际到账金额给钱包入账。
/
REJECTED（已拒绝） / CANCELED（已取消）后台拒绝或用户取消，不产生钱包入账。

8.2 充值状态与触发

状态 业务含义 进入条件 可去向 资金影响
SUBMITTED（已提交） 用户已提交，等待审核。 用户创建充值订单成功。 APPROVED（已审核通过）、REJECTED（已拒绝）、CANCELED（已取消）。 无资金变化。
APPROVED（已审核通过） 后台确认充值有效。 管理员审核通过。 终态。 钱包可用余额增加，累计充值增加，写 RECHARGE 流水。
REJECTED（已拒绝） 后台确认充值无效。 管理员审核拒绝。 终态。 无资金变化。
CANCELED（已取消） 用户主动取消。 用户取消 SUBMITTED（已提交） 充值单。 终态。 无资金变化。

8.3 充值校验规则

  • 充值渠道必须启用。
  • 充值金额必须大于 0。
  • 充值金额必须不低于系统最低充值金额和渠道最低充值金额的较大值。
  • 如果用户提交外部交易号，当前会检查重复，避免同一链上交易号重复提交。
  • 创建充值订单时会加用户级 Redis 锁，减少重复提交。
业务缺口确认：当前充值是人工审核模型，没有看到链上自动回调入账流程。若产品要求自动到账，需要新增链上确认、回调幂等、异常补偿和审核兜底规则。

9.2 创建订单

用户先选择商品、AI 模型和租赁周期，然后创建租赁订单。创建时不会扣钱，只会生成 PENDING_PAY（待支付） 订单，并保存商品、模型、周期的快照字段。

检查项 当前规则
商品 商品必须存在且状态 ENABLED（启用）。
AI 模型 AI 模型必须存在且状态 ENABLED（启用）。
租赁周期 周期规则必须存在且状态 ENABLED（启用）。

9.3 支付机器费

PENDING_PAY（待支付）订单已创建但未支付。
→
钱包扣机器费扣除 orderAmount，写 RENT_PAY（租赁支付） 流水。
→
生成 API 凭证凭证状态 GENERATED（已生成），保存加密 token 和脱敏 token。
→
PENDING_ACTIVATION（待支付部署费）等待用户支付部署费。

支付机器费成功后，订单的 paidAmount 等于订单金额，记录 paidAt 和 apiGeneratedAt。如果用户余额不足，支付失败，订单仍保持 PENDING_PAY（待支付）。

9.4 待激活阶段的取消和超时

场景 允许状态 触发方式 结果
用户取消未付款订单 PENDING_PAY（待支付） 用户点击取消。 订单 CANCELED（已取消），不涉及退款。
用户取消已付机器费订单 PENDING_ACTIVATION（待支付部署费） 用户点击取消。 订单 CANCELED（已取消），机器费退回钱包，API 凭证 REVOKED（已撤销）。
系统超时取消 PENDING_ACTIVATION（待支付部署费） 定时任务每 10 分钟扫描，默认 60 分钟未支付部署费。 订单 CANCELED（已取消），机器费退回钱包，API 凭证 REVOKED（已撤销）。

9.5 支付 API 部署费

PENDING_ACTIVATION（待支付部署费）机器费已付，API 凭证 GENERATED（已生成）。
→
钱包扣部署费扣除 AI 模型部署费，写 API_DEPLOY_FEE（API部署费） 流水。
→
部署订单 PAID（已支付）创建 api_deploy_order 并立即置为 PAID（已支付）。
→
API ACTIVE（生效中）API 凭证立即生效。
→
订单 RUNNING（运行中）设置 startedAt、profitStartAt、profitEndAt 和 autoPauseAt。
当前期排除：部署订单状态定义包含 PENDING_PAY（待支付）、PAID（已支付）、CANCELED（已取消）、REFUNDED（已退款）。当前上线口径只使用 PENDING_PAY（待支付）→ PAID（已支付），部署订单取消/退款流程暂不纳入本期。

9.6 激活后自动暂停与用户启动

RUNNING（运行中）部署费已付，订单立即开始运行并写入收益周期。
→
运行片段开始记录 segmentStartAt，后续收益只按片段分钟计算。
→
自动暂停每 5 分钟扫描 autoPauseAt 到期的订单，关闭运行片段。
→
PAUSED（已暂停）订单和 API 凭证都暂停。
→
用户再次启动用户手动 start，开启新的运行片段。
→
RUNNING（运行中）租期不重置、不延长，到 profitEndAt 结束。

用户支付部署费成功后，系统把订单直接置为 RUNNING（运行中），设置 startedAt、profitStartAt 和 profitEndAt。收益结束时间 = 部署费支付成功时间 + 周期天数。自动暂停不会延长租期；用户从 PAUSED（已暂停）再次启动时只开启新的运行片段，原 profitEndAt 不变。

API 凭证展示口径：API Token 当前仅作为页面展示用的模拟凭证。系统保存加密 token 和脱敏 token，前台只展示脱敏信息，不向用户交付可复制调用的明文 token；本期测试不把明文 token 交付作为业务闭环要求。

9.7 租赁订单状态说明

状态 产品含义 进入触发 允许动作/后续
PENDING_PAY（待支付） 订单已创建，机器费未支付。 用户创建订单。 用户支付机器费；用户取消。
PAID（已支付） 枚举保留状态。 当前主流程未使用该状态。 前端枚举也未暴露。
PENDING_ACTIVATION（待支付部署费） 机器费已付，API 凭证已生成，等待支付部署费。 用户支付机器费成功。 用户支付部署费；用户取消；超时自动取消。
ACTIVATING（激活中） 枚举保留/兼容状态。 当前部署费支付成功后主流程直接进入 RUNNING（运行中）。 自动暂停任务仍兼容扫描该状态。
PAUSED（已暂停） 订单暂停中，暂停期间不产生收益，但租期继续消耗。 自动暂停任务；后台禁用用户。 未到期时用户可启动；已到期则等待系统到期结算；用户也可提前结算。
RUNNING（运行中） 订单运行中，每日收益可产生。 部署费支付成功；或用户从 PAUSED（已暂停）启动。 每日收益；到期结算；提前结算；后台禁用用户会暂停。
EXPIRED（已过期） 订单到期完成。 到期结算任务。 终态。
SETTLING（结算中） 结算中短暂状态。 到期或提前结算过程中。 很快变为 EXPIRED（已过期） 或 EARLY_CLOSED（提前关闭）。
SETTLED（已结算） 枚举保留状态。 当前主流程未看到实际落该状态。 前端枚举未暴露。
EARLY_CLOSED（提前关闭） 用户提前结算关闭。 用户对 RUNNING（运行中） 或 PAUSED（已暂停） 订单提前结算。 终态。
CANCELED（已取消） 订单取消。 用户取消 PENDING_PAY（待支付）/PENDING_ACTIVATION（待支付部署费）；或部署费超时未支付。 终态。
已校准口径：部署费支付后订单立即 RUNNING（运行中）并开始租期；24 小时自动暂停只是平台粘性机制，暂停期间没有收益但租期继续消耗，用户再次启动不会重算或延长周期。

10. 每日收益业务

10.1 收益什么时候开始

收益不是从创建订单或支付机器费时开始，而是从用户支付部署费成功、订单进入 RUNNING（运行中） 时开始。系统会写入 profitStartAt 和 profitEndAt，并开启第一段运行片段；后续暂停再启动会开启新的运行片段，但不会改变租期结束时间。

部署费支付成功订单直接进入 RUNNING（运行中）。
→
开启运行片段记录真实运行起点。
→
暂停则关闭片段PAUSED（已暂停） 不产收益，租期继续走。
→
每日 00:00结算上一自然日的运行分钟收益。
→
收益入账钱包可用余额增加，收益记录 SETTLED（已结算）。

10.2 收益记录状态

状态 说明 进入触发 资金影响
PENDING（待处理） 收益记录创建但尚未入账。 每日收益任务或到期结算任务补结当天收益时创建记录。 短暂状态，尚未增加钱包。
SETTLED（已结算） 收益已入账。 钱包成功 credit。 用户钱包可用余额增加，累计租赁收益增加。
CANCELED（已取消） 枚举存在。 当前未看到主流程使用。 无当前业务说明。

10.3 每日收益计算

字段 来源 说明
tokenOutput 订单快照：商品每日 token 产出。 创建订单时从商品复制。
tokenUnitPrice 订单快照：AI 模型 token 单价。 创建订单时从 AI 模型复制。
yieldMultiplier 订单快照：周期收益倍数。 创建订单时从周期规则复制。
完整日收益 tokenOutput * tokenUnitPrice * yieldMultiplier 完整运行 1440 分钟的理论日收益，也是前台预估日收益口径。
effectiveMinutes 运行片段在该自然日内的完整分钟数汇总。 只计算 RUNNING（运行中） 片段的完整分钟；PAUSED（已暂停） 时间不计收益，不足 1 分钟不计。
finalProfit 完整日收益 / 1440 * effectiveMinutes 实际入账金额。有效分钟为 0 时不生成收益记录。
当前收益口径：每日收益任务每天 00:00 后结算刚刚结束的上一自然日收益；收益按运行片段汇总完整分钟。订单到期或用户提前结算时，如果当前自然日还有未结算运行分钟，会先补结到截止时间，再返还本金并关闭订单。

10.4 哪些订单不会产生收益

  • 未进入运行的订单：PENDING_PAY（待支付）、PENDING_ACTIVATION（待支付部署费）、ACTIVATING（激活中）。
  • PAUSED（已暂停） 期间：订单暂停不产收益，但租期继续消耗；暂停前已经形成的运行片段仍会在每日收益或最终结算时入账。
  • 已结束的订单：EXPIRED（已过期）、EARLY_CLOSED（提前关闭）、CANCELED（已取消）。
  • 待结算自然日没有任何运行片段交集的订单。
  • 当天有效运行时长不足 1 个完整分钟的订单。
  • 同一订单同一收益日期已经生成过收益记录的订单。

11. 佣金与团队业务

11.1 佣金从哪里开始

佣金不是从用户充值、支付订单或提现时产生，而是从“下级用户每日租赁收益已入账”开始。佣金任务每 5 分钟扫描已经 SETTLED（已结算） 且未生成佣金的收益记录，然后按邀请链给一级、二级上级发佣金。

下级收益 SETTLED（已结算）下级每日收益已经进入钱包。
→
查邀请链只查一级、二级上级。
→
取佣金规则优先使用启用规则，无规则时使用默认比例。
→
佣金入账给上级钱包增加可用余额。
→
标记已生成收益记录 commissionGenerated = 1。

11.2 佣金比例

层级 默认比例 来源
一级 20% 启用的 commission_rule；无配置时默认 20%。
二级 10% 启用的 commission_rule；无配置时默认 10%。
佣金统计口径：当前佣金枚举、佣金规则、用户佣金列表、后台佣金列表都只支持 1、2 两个层级。用户佣金汇总不再直接读取钱包的累计佣金字段，而是按已结算佣金记录汇总一级和二级金额。

11.3 佣金状态

状态 说明 触发 资金变化
PENDING（待处理） 佣金记录创建但尚未入账。 佣金任务创建记录。 短暂状态，尚未增加钱包。
SETTLED（已结算） 佣金已经入账。 上级钱包 credit 成功。 上级可用余额增加，累计佣金增加。
CANCELED（已取消） 枚举存在。 当前未看到主流程使用。 无当前业务说明。
产品口径：佣金只基于下级每日租赁收益生成，不基于充值、购买机器费、部署费、提现或本金结算生成。

12. 到期/提前结算

12.1 到期结算

到期结算由系统定时任务触发。任务每 1 分钟扫描 RUNNING（运行中） 或 PAUSED（已暂停） 且收益结束时间已到的订单；命中后先按运行片段补结尚未结算的完整分钟收益，再把订单改为结算中，创建结算单，返还机器租赁本金，最后把订单置为 EXPIRED（已过期）。

RUNNING（运行中） / PAUSED（已暂停）profitEndAt 已到，暂停订单也要到期结算。
→
补结未结收益按运行片段完整分钟补结到 profitEndAt。
→
SETTLING（结算中）订单进入结算中，避免重复结算。
→
创建结算单type = EXPIRE（到期结算），status = SETTLED（已结算）。
→
返还本金actualSettle = orderAmount，钱包入账 SETTLEMENT。
→
EXPIRED（已过期）订单完成，API 凭证 EXPIRED（已过期）。

12.2 提前结算

用户可对 RUNNING（运行中） 或 PAUSED（已暂停） 订单发起提前结算。提前结算会先关闭当前运行片段并按完整分钟补结到当前时间，再按照订单快照中的罚金比例计算罚金，并返还“本金 - 罚金”。罚金当前以无余额变化的流水记录下来，不会再从用户钱包额外扣款。

RUNNING（运行中） / PAUSED（已暂停）只有这两个状态可提前结算。
→
SETTLING（结算中）订单进入结算中。
→
计算罚金penalty = orderAmount * earlyPenaltyRate。
→
返还剩余本金actualSettle = orderAmount - penalty。
→
EARLY_CLOSED（提前关闭）订单关闭，API 凭证 REVOKED（已撤销）。

12.3 结算单状态

状态 含义 当前使用情况
PENDING（待处理） 待处理。 枚举存在，当前到期和提前结算都直接创建 SETTLED（已结算）。
SETTLED（已结算） 已结算。 当前主流程使用，到期和提前结算都直接完成。
REJECTED（已拒绝） 已拒绝。 枚举存在，当前未看到后台审核拒绝流程。
CANCELED（已取消） 已取消。 枚举存在，当前未看到主流程使用。
产品确认：结算类型包含 EXPIRE（到期结算）、EARLY_TERMINATE（提前终止）、MANUAL（人工处理），但当前只看到到期和提前结算入口；MANUAL（人工处理） 人工结算、结算审核、拒绝、取消当前未形成业务流程。

13. 提现业务

13.1 提现从哪里开始

提现从用户提交提现申请开始。系统先校验金额、每日限额、提现地址格式和钱包余额，然后把申请金额从可用余额转入冻结余额。后台审核通过后，管理员再确认已打款；打款确认后，冻结余额被扣减，提现闭环。

用户申请提现校验金额、日限额、地址、余额。
→
冻结资金可用余额减少，冻结余额增加。
→
PENDING_REVIEW（待审核）等待后台审核，用户可取消。
→
APPROVED（已审核通过）后台审核通过，等待打款。
→
PAID（已支付）后台确认打款，扣减冻结余额。
PENDING_REVIEW（待审核）待审核。
→
CANCELED（已取消）用户取消，冻结资金解冻。
/
REJECTED（已拒绝）后台拒绝，冻结资金解冻。

13.2 提现状态与资金变化

状态 业务含义 进入触发 后续 资金变化
PENDING_REVIEW（待审核） 提现待后台审核。 用户创建提现申请成功。 用户取消、后台通过、后台拒绝。 申请金额已从可用转冻结。
APPROVED（已审核通过） 后台审核通过，等待实际打款。 管理员审核通过。 后台确认打款；后台仍可拒绝。 资金仍在冻结中。
PAID（已支付） 后台已确认打款。 管理员填写打款信息并确认 paid。 终态。 冻结余额扣减；累计提现按实际到账金额增加。
REJECTED（已拒绝） 后台拒绝提现。 管理员拒绝 PENDING_REVIEW（待审核） 或 APPROVED（已审核通过） 订单。 终态。 冻结资金解冻回可用余额。
CANCELED（已取消） 用户取消提现。 用户取消 PENDING_REVIEW（待审核） 订单。 终态。 冻结资金解冻回可用余额。

13.3 手续费和到账金额

规则 当前实现 举例
免手续费 申请金额大于等于免手续费门槛时，手续费为 0。 门槛 100，申请 100，手续费 0。
收手续费 申请金额低于门槛时，手续费 = 申请金额 * 手续费率。 门槛 100，费率 5%，申请 50，手续费 2.5，到账 47.5。
冻结金额 冻结用户申请金额，而不是实际到账金额。 申请 50 时冻结 50。
累计提现 打款成功后累计提现增加实际到账金额。 申请 50，到账 47.5，则累计提现增加 47.5。

13.4 提现地址校验

  • TRC20 地址必须以 T 开头并符合 Tron 地址长度和字符规则。
  • ERC20/BEP20 等 EVM 网络地址必须符合 0x 加 40 位十六进制格式。
  • 网络名称会标准化后再校验。

14.1 后台能力总览

后台模块 能做什么 对前台业务的影响
管理员认证与账号 后台登录、注册管理员、管理管理员账号、查看登录日志。 决定谁能进入后台。

14.2 用户禁用/启用

后台禁用用户app_user.status = DISABLED（禁用）。
→
暂停运行订单RUNNING（运行中）订单改 PAUSED（已暂停），profitStatus 改 PAUSED（已暂停）。
→
关闭运行片段关闭当前 segment，禁用后的时间不再算收益。
→
暂停 APIACTIVE（生效中） API 凭证改 PAUSED（已暂停）。
→
记录日志后台操作写 sys_admin_log。

当前业务口径为：用户一旦被后台禁用，不再重新启用。后台启用接口保留但会返回业务错误，不会恢复用户状态、订单或 API。

14.3 定时任务

任务 默认时间/频率 作用 失败影响
部署费超时取消 每 10 分钟。 取消 PENDING_ACTIVATION（待支付部署费） 且超时未付部署费的订单，退机器费。 订单可能长期卡在 PENDING_ACTIVATION（待支付部署费）。
自动暂停 每 5 分钟扫描；自动暂停时间是部署费支付后 24 小时。 把 RUNNING（运行中）/ACTIVATING（激活中） 且 autoPauseAt 到期的订单/API 改 PAUSED（已暂停），并关闭当前运行片段。 暂停期间不产生收益，但租期继续消耗。
每日收益 每天 00:00。 按运行片段给上一自然日有完整运行分钟的订单生成收益并入账，PAUSED（已暂停） 订单的历史运行片段不会漏结。 用户收益不到账，佣金也不会生成。
到期结算 每 1 分钟。 RUNNING（运行中）/PAUSED（已暂停） 到期后补结未结运行分钟收益，返还本金并置为 EXPIRED（已过期）。 订单超过收益期仍未关闭，本金未返还。
佣金生成 每 5 分钟。 给已结算收益的上级发佣金。 上级佣金延迟或不入账。

18. 上线校准清单

以下是从代码实现反推出来的产品校准点。建议产品、后端、测试、运营一起逐项确认：如果符合需求，就写入上线口径；如果不符合，需要排期调整。

  1. 商品库存与可租日期：已确认当前只作为展示/运营参考，不参与下单交易校验；下单只校验商品、AI 模型、周期是否启用。
  2. 自动暂停业务口径：用户支付部署费后订单立即 RUNNING（运行中） 并开始租期，24 小时后自动 PAUSED（已暂停）。暂停期间不产生收益但租期继续消耗，这是已确认的平台粘性机制。
  3. 每日收益分钟口径：当前已按完整分钟结算上一自然日收益，不足 1 分钟不计；需要前端和用户说明同步“预估日收益是 1440 分钟完整日收益，实际按有效分钟折算”。
  4. 结算审核缺失：结算单有待处理、拒绝、取消状态，但实际流程直接 SETTLED（已结算）。若运营需要审核结算，当前未实现。
  5. 人工结算缺失：结算类型有 MANUAL（人工处理），但当前未看到人工结算入口。异常订单如何结算需要明确。
  6. 部署订单取消/退款：当前期排除。部署订单本期只走 PENDING_PAY（待支付）→ PAID（已支付），CANCELED（已取消）/REFUNDED（已退款） 暂不实现业务路径。
  7. 自动通知：下一期实现。本期不要求收益、佣金、订单到期等业务事件自动发通知。
  8. 禁用用户后的业务处理：已闭环。禁用会暂停 RUNNING（运行中）订单、关闭运行片段并暂停 ACTIVE（生效中）API；旧 JWT 会被后端实时拒绝；用户禁用后不可重新启用。
  9. 充值模式：当前是人工审核入账，未看到链上自动确认或支付回调。需要确认运营审核工作量和到账时效。
  10. 提现手续费口径：申请金额达到免手续费门槛时手续费为 0，未达到则按申请金额乘费率。累计提现统计实际到账金额，不含手续费。
  11. 提现打款责任边界：系统只在管理员确认 PAID（已支付） 后扣冻结余额，实际链上打款是否成功依赖管理员录入和外部流程。
  12. 代理层级已调整为二级：当前邀请关系、团队关系、佣金规则、用户查询和后台查询都只保留一级、二级。若历史库里存在三级团队或三级佣金数据，需要确认迁移、隐藏、历史账务展示和钱包累计佣金口径。
  13. 后台调账：钱包业务类型有 ADJUST（调账），但未看到人工调账接口。若需要财务纠错能力，需要补审批和审计。
  14. API token 展示：已确认 API Token 仅作为页面模拟凭证展示。系统保存加密 token 和脱敏 token，前台只展示脱敏信息，不向用户交付可复制调用的明文 token。
  15. 初始化数据风险：若商品 token 产出、AI token 单价或部署费配置为 0，则预计收益、每日收益或部署费会是 0。上线前必须核查生产配置。
  16. 定时任务监控：收益、佣金、自动暂停、到期结算都依赖任务执行。需要上线监控和失败补偿 SOP。

18.1 建议产品校准会议顺序

顺序 主题 建议参与人 输出结论
1 商品售卖规则 产品、运营、后端、测试 已确认库存和可租日期仅展示；交易只校验启用状态和价格快照。
2 租赁订单生命周期 产品、后端、测试 每个状态是否符合需求，哪些状态允许用户取消或退款。
3 收益与结算口径 产品、财务、后端、测试 收益日期、收益金额、到期本金返还、提前罚金、人工结算。
4 充值提现财务流程 产品、财务、运营、后端 审核职责、到账时间、手续费、异常处理、链上凭证规则。
5 邀请佣金 产品、运营、后端 佣金来源、一级/二级范围、比例、发放时间、异常追回规则。
6 消息与运营后台 产品、运营、客服、后端 自动通知进入下一期；本期确认后台手动消息和看板是否满足运营使用。
上线校准目标：每一条业务状态都要能回答四个问题：谁触发、触发条件是什么、资金是否变化、失败或异常时怎么补偿。只要有一个问题答不上来，就应该视为上线前待确认项。
