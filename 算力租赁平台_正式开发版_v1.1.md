## 2. 已确认的核心业务规则

### 2.1 平台业务主线

```text
用户输入邮箱
→ 发送验证码（每分钟最多5次，Redis频率限制）
→ 验证码校验通过
→ 邮箱不存在则自动注册，存在则直接登录
→ 钱包初始化（注册时同步创建，同一事务）
→ 推荐关系和团队关系写入（注册时同步写入，同一事务）
→ 充值 USDT（最低500 USDT，客服审核入账）
→ 选择 GPU 机器产品
→ 选择 AI API 模型
→ 选择租赁周期
→ 查看预估收益
→ 下单并支付机器费用
→ 后端生成 API Token
→ 用户在控制台点击激活并支付 API 部署费
→ 支付成功后直接进入激活中（ACTIVATING），无需再次确认
→ 24 小时后系统自动改为暂停（PAUSED）
→ 用户点击启动
→ 订单变为运行中（RUNNING）
→ 从启动时间开始产生每日收益
→ 到期结算或提前结算
→ 提现 USDT（最低10 USDT）
```

### 2.2 关键业务结论

```text
机器费用支付成功 ≠ 开始收益。
API Token 生成 ≠ 开始收益。
API 部署费支付成功 = 直接进入 ACTIVATING（无需再次点击激活）。
API 激活/部署中（ACTIVATING）≠ 开始收益。
24 小时后自动暂停 ≠ 开始收益。
只有用户点击启动，并且订单状态为 RUNNING 后，才开始产生每日收益。
```

### 2.3 收益公式

第一版收益公式固定为：

```text
每日收益 = GPU每日产出Token × 所选AI模型Token单价 × 周期收益倍率
```

说明：

1. 本平台不记录用户真实 Token 调用量。
2. Token 月消耗量、单个 Token 价格、模拟 Token 数据主要用于前台展示。
3. 真实 Token 消耗由其他平台或人工体系管理。
4. 本平台的重点是：用户购买机器后能看到预估收益、每日收益、累计收益、可提现余额。
5. 历史订单必须保存产品、模型、周期、收益规则快照，避免后台改规则后影响旧订单。

### 2.4 资金规则

```text
全平台统一使用 USDT。
充值使用 USDT。
购买机器使用 USDT。
API 部署费用使用 USDT。
每日收益使用 USDT。
推广佣金使用 USDT。
提现使用 USDT。
```

金额字段统一使用：

```sql
DECIMAL(20, 8)
```

所有金额计算必须使用 BigDecimal，禁止使用 double/float。

### 2.5 产品库存规则

```text
产品库存只作为后台手动展示字段。
不做真实库存锁。
不防超买超卖。
后台想增加多少产品、改多少库存都可以。
下单时不做复杂库存扣减事务。
```

### 2.6 API Token 展示规则

```text
api_credential 只表示订单对应的 API 凭证。
API 数据用于前台控制台展示和模拟变化。
平台不记录真实 Token 调用明细。
不建立 api_token_usage_daily 表。
不在 api_credential 中保存真实用量字段。
```

### 2.7 提现规则（v1.1新增）

```text
最低提现金额：10 USDT（sys_config 可配置）
手续费规则：
  提现金额 >= 100 USDT → 手续费 = 0（免手续费）
  提现金额 <  100 USDT → 手续费 = 提现金额 × 5%
免手续费门槛（100）和手续费比例（5%）均从 sys_config 读取，后台可配置。
每日累计提现上限：100000 USDT（sys_config 可配置）
  校验范围：当天 status IN (PENDING_REVIEW, APPROVED, PAID) 的提现单合计
无单笔最高提现上限限制。
提现地址格式校验：
  TRC20: ^T[1-9A-HJ-NP-Za-km-z]{33}$
  ERC20/BEP20: ^0x[0-9a-fA-F]{40}$
  前端和后端均做校验，后端校验不通过直接拒绝。
提现必须走冻结余额流程（见 2.8）。
```

### 2.8 提现冻结规则

```text
用户申请提现：
  校验 apply_amount >= withdraw.min_amount（10）
  校验今日累计 + apply_amount <= withdraw.max_daily_amount（100000）
  校验提现地址格式
  计算手续费：apply_amount >= fee_free_threshold(100) ? 0 : apply_amount × fee_rate(0.05)
  actual_amount = apply_amount - fee_amount
  available_balance 减少 apply_amount
  frozen_balance 增加 apply_amount
  生成 withdraw_order
  写 wallet_transaction（tx_type=FREEZE）

审核通过并打款：
  frozen_balance 减少 apply_amount
  total_withdraw 增加 actual_amount
  withdraw_order.status = PAID
  写 wallet_transaction（tx_type=OUT）

审核驳回：
  frozen_balance 减少 apply_amount
  available_balance 增加 apply_amount
  withdraw_order.status = REJECTED
  写 wallet_transaction（tx_type=UNFREEZE）
```

### 2.9 充值规则（v1.1新增）

```text
全局最低充值金额：500 USDT（sys_config 可配置）
渠道可单独设 min_amount，校验时取 MAX(全局最低, 渠道最低)。
充值走客服审核流程，审核通过后入账。
充值单的 external_tx_no（交易哈希）做唯一约束，防止同一笔链上交易重复入账。
```

### 2.10 三级佣金与团队归属规则

```text
团队归属可以无限向下统计。
佣金只发三级（基于 user_referral_relation）。
一级佣金 20%。（注意：commission_rule 表初始化数据以本节为准）
二级佣金 10%。
三级佣金 5%。
四级及更深层级只属于团队统计，不产生佣金。
```

示例：

```text
A
└── B
    └── C
        └── D
            └── E
```

对于 A：

```text
B 是 A 的一级团队成员，B 产生收益时，A 拿 20%。
C 是 A 的二级团队成员，C 产生收益时，A 拿 10%。
D 是 A 的三级团队成员，D 产生收益时，A 拿 5%。
E 是 A 的四级团队成员，E 仍属于 A 的团队统计，但 A 不拿 E 的佣金。
```

收益触发逻辑：

```text
用户 X 产生收益。
X 的直属上级拿 20%。
X 的上上级拿 10%。
X 的上上上级拿 5%。
再往上不发佣金。
```

佣金发放顺序：收益先入账给产生收益的用户，入账成功后再触发三级佣金计算（分离事务，避免佣金失败导致收益回滚）。

### 2.11 用户注销/封号规则（v1.1新增）

```text
后台封号：app_user.status = 0
封号触发：
  扫描该用户所有 order_status = RUNNING 的订单，批量改为 PAUSED
  api_credential.token_status 同步改为 PAUSED
  操作写入 sys_admin_log
封号后已冻结的提现单：保持原状，由后台人工处理
封号不退还已支付的机器费用和部署费用
```

### 2.12 订单取消规则（v1.1新增）

```text
PENDING_PAY 状态可取消：
  机器费未扣，直接取消，无需退款
  rental_order.order_status = CANCELED

PENDING_ACTIVATION 状态可取消：
  退还机器费（order_amount）到 user_wallet.available_balance
  写 wallet_transaction（biz_type=REFUND）
  api_credential.token_status = REVOKED
  rental_order.order_status = CANCELED
  写 sys_notification 通知用户

PENDING_ACTIVATION 超时自动取消（1小时，sys_config 可配置）：
  定时任务扫描：order_status = PENDING_ACTIVATION AND api_generated_at <= now() - 60分钟
  执行同上退款逻辑
  发送超时取消通知

部署费已支付后（ACTIVATING 及之后）：
  不支持普通取消，只支持提前结算（见 2.13）
  部署费不退还（因为已直接进入激活流程）
```

### 2.13 提前结算规则

```text
用户未到期申请提前结算：
  生成 rental_settlement_order，settlement_type = EARLY_TERMINATE
  penalty_amount = order_amount × early_penalty_rate_snapshot（默认1%）
  actual_settle_amount = order_amount - penalty_amount（本金扣违约金后退还）
  写 wallet_transaction（biz_type=EARLY_PENALTY 扣违约金）
  写 wallet_transaction（biz_type=SETTLEMENT 退本金）
  rental_order.order_status = EARLY_CLOSED
  api_credential.token_status = REVOKED
```

### 2.14 收益与结算边界规则（v1.1新增）

```text
每日收益任务扫描条件：
  order_status = RUNNING
  AND profit_start_at <= 当前时间
  AND profit_date < DATE(profit_end_at)   ← 最后一天不在此任务发放
到期结算任务单独处理：
  当前时间 >= profit_end_at
  → order_status = EXPIRED
  → 生成 rental_settlement_order
两个任务互不干扰，通过 profit_date 边界严格区分。
```

### 2.15 时区规范（v1.1新增）

```text
服务器部署在香港（UTC+8），用户主要在中国大陆（UTC+8），时区一致。
所有 DATETIME 字段存储和写入均使用 UTC+8（Asia/Shanghai）。
profit_date 按 UTC+8 日期边界（每日0点）切割。
代码层时区统一设置为 Asia/Shanghai，禁止使用服务器默认时区。
```

### 2.16 登录方式（v1.1确认）

```text
当前版本：纯邮箱验证码登录，注册和登录合一。
  - 邮箱存在 → 直接登录
  - 邮箱不存在 → 自动注册后登录
app_user.password_hash 字段保留（为后续扩展密码登录预留），当前不使用。
email_verify_code.scene 当前只使用 LOGIN，RESET_PASSWORD 暂不启用。
```

# 5. 完整表结构

## 5.1 app_user｜前台用户表

代表业务：前台会员账号。用户通过邮箱验证码注册或登录（注册和登录合一）。新用户注册成功后，必须在同一事务内创建钱包、推荐关系和团队关系。`password_hash` 字段保留供后续扩展，当前版本不使用。

```sql
CREATE TABLE `app_user` (
  `id`                BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `user_id`           VARCHAR(64) NOT NULL COMMENT '用户编号，业务唯一标识',
  `email`             VARCHAR(128) NOT NULL COMMENT '邮箱，唯一',
  `password_hash`     VARCHAR(255) DEFAULT NULL COMMENT '登录密码哈希，当前版本保留不使用',
  `user_name`          VARCHAR(64) DEFAULT NULL COMMENT '用户昵称',
  `status`            TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1-正常，0-禁用',
  `email_verified_at` DATETIME DEFAULT NULL COMMENT '邮箱验证时间',
  `last_login_at`     DATETIME DEFAULT NULL COMMENT '最后登录时间',
  `created_at`        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_id` (`user_id`),
  UNIQUE KEY `uk_email` (`email`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='前台用户表';
```

## 5.2 email_verify_code｜邮箱验证码表

代表业务：邮箱登录、注册验证码校验。验证码只存哈希，不存明文。发送频率限制（每分钟最多5次）由 Redis 控制，不在此表记录。当前版本 scene 只使用 LOGIN。

```sql
CREATE TABLE `email_verify_code` (
  `id`         BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `email`      VARCHAR(128) NOT NULL COMMENT '邮箱',
  `scene`      VARCHAR(32) NOT NULL DEFAULT 'LOGIN' COMMENT '场景：LOGIN-登录/注册（当前版本），RESET_PASSWORD-重置密码（暂不启用）',
  `code_hash`  VARCHAR(255) NOT NULL COMMENT '验证码哈希',
  `send_ip`    VARCHAR(64) DEFAULT NULL COMMENT '发送IP',
  `expire_at`  DATETIME NOT NULL COMMENT '过期时间',
  `used_at`    DATETIME DEFAULT NULL COMMENT '使用时间',
  `status`     TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-未使用，1-已使用，2-已过期',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  PRIMARY KEY (`id`),
  KEY `idx_email_scene` (`email`, `scene`),
  KEY `idx_expire_at` (`expire_at`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='邮箱验证码表';
```

**Redis 频率限制设计：**

```text
Key:   email_code_rate:{email}:{scene}
Value: 发送次数（整数）
TTL:   60秒（自动过期）
逻辑：发送前 INCR，值 > 5 则拒绝；第一次发送时 SET key 1 EX 60 NX（原子操作）
```

## 5.3 user_referral_relation｜用户推荐关系表

代表业务：保存用户的直属上级、三级返佣上级和自己的邀请码。该表用于佣金发放，佣金只发三级。注册时与 app_user、user_wallet 在同一事务内写入。

```sql
CREATE TABLE `user_referral_relation` (
  `id`                 BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `user_id`            BIGINT NOT NULL COMMENT '当前用户ID',
  `invite_code`        VARCHAR(32) NOT NULL COMMENT '当前用户自己的邀请码',
  `parent_user_id`     BIGINT DEFAULT NULL COMMENT '直属上级用户ID',
  `parent_invite_code` VARCHAR(32) DEFAULT NULL COMMENT '注册时使用的邀请码',
  `level1_user_id`     BIGINT DEFAULT NULL COMMENT '一级返佣上级（直属上级），收益时拿20%',
  `level2_user_id`     BIGINT DEFAULT NULL COMMENT '二级返佣上级，收益时拿10%',
  `level3_user_id`     BIGINT DEFAULT NULL COMMENT '三级返佣上级，收益时拿5%',
```

## 5.5 user_wallet｜用户钱包表

代表业务：每个用户一个 USDT 钱包。钱包保存可用余额、冻结余额、累计充值、累计提现、累计收益、累计佣金。`version_no` 为乐观锁，所有余额变更必须携带版本号，更新行数为0时抛出并发冲突异常。

```sql
CREATE TABLE `user_wallet` (
  `id`                BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `wallet_no`         VARCHAR(64) NOT NULL COMMENT '钱包编号',
  `user_id`           BIGINT NOT NULL COMMENT '用户ID',
  `currency`          VARCHAR(10) NOT NULL DEFAULT 'USDT' COMMENT '币种，固定USDT',
  `available_balance` DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '可用余额',
  `frozen_balance`    DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '冻结余额',
  `total_recharge`    DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '累计充值成功金额',
  `total_withdraw`    DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '累计提现成功金额',
  `total_profit`      DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '累计租赁收益金额',
  `total_commission`  DECIMAL(20,8) NOT NULL DEFAULT 0.00000000 COMMENT '累计推广佣金金额',
  `status`            TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1-正常，0-停用',
  `version_no`        INT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号，所有余额变更必须带此字段',
  `created_at`        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at`        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_wallet_no` (`wallet_no`),
  UNIQUE KEY `uk_user_id` (`user_id`),
  KEY `idx_status` (`status`),

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户USDT钱包表';
```

**乐观锁扣款示例：**

```sql
UPDATE user_wallet
SET available_balance = available_balance - ?,
    version_no = version_no + 1
WHERE id = ? AND version_no = ? AND available_balance >= ?
```

更新行数为0时，需判断是并发冲突还是余额不足，并向上层抛出对应异常。

# 7. 枚举字典

## 7.1 通用状态

| 字段 | 值 | 含义 |
|---|---:|---|
| status | 1 | 启用/正常/有效 |
| status | 0 | 停用/禁用/无效 |

## 7.2 email_verify_code.scene

| 值 | 含义 | 当前状态 |
|---|---|---|
| LOGIN | 登录/注册验证码 | 启用 |
| RESET_PASSWORD | 重置密码验证码 | 暂不启用 |

## 7.3 wallet_transaction.tx_type

| 值 | 含义 | 余额影响 |
|---|---|---|
| IN | 入账 | 可用余额增加 |
| OUT | 支出 | 可用余额减少或冻结余额减少 |
| FREEZE | 冻结 | 可用余额减少，冻结余额增加 |
| UNFREEZE | 解冻 | 冻结余额减少，可用余额增加 |

## 7.4 wallet_transaction.biz_type

| 值 | 含义 | 典型来源表 |
|---|---|---|
| RECHARGE | 充值 | recharge_order |
| WITHDRAW | 提现 | withdraw_order |
| RENT_PAY | 购买/租赁机器扣费 | rental_order |
| API_DEPLOY_FEE | API部署费用 | api_deploy_order |
| RENT_PROFIT | 租赁每日收益 | rental_profit_record |
| COMMISSION_PROFIT | 推广佣金 | commission_record |
| SETTLEMENT | 订单结算（本金退还） | rental_settlement_order |
| EARLY_PENALTY | 提前解约违约金 | rental_settlement_order |
| REFUND | 退款（超时取消/人工） | rental_order/人工处理 |
| ADJUST | 调账（仅SUPER_ADMIN） | 后台人工处理 |

## 7.5 rental_order.order_status

| 值 | 含义 | 是否产生收益 | 可取消 |
|---|---|---|---|
| PENDING_PAY | 待支付机器费用 | 否 | 是（无需退款） |
| PAID | 机器费用已支付 | 否 | — |
| PENDING_ACTIVATION | API已生成，待用户激活 | 否 | 是（退机器费） |
| ACTIVATING | 已支付部署费，API激活中 | 否 | 否（只支持提前结算） |
| PAUSED | 激活24小时后自动暂停，等待用户启动 | 否 | 否 |
| RUNNING | 用户已启动，正在运行 | 是 | 否（只支持提前结算） |
| EXPIRED | 已到期 | 否 | — |
| SETTLING | 结算中 | 否 | — |
| SETTLED | 已结算 | 否 | — |
| EARLY_CLOSED | 提前解约关闭 | 否 | — |
| CANCELED | 已取消 | 否 | — |

## 7.6 api_credential.token_status

| 值 | 含义 |
|---|---|
| GENERATED | API已生成，未激活 |
| ACTIVATING | 激活/部署中（支付部署费后直接进入） |
| PAUSED | 已暂停，等待用户启动 |
| ACTIVE | 运行中 |
| EXPIRED | 已过期 |
| REVOKED | 已吊销（取消/提前结算） |

## 7.7 rental_order.profit_status

| 值 | 含义 |
|---|---|
| NOT_STARTED | 未开始收益 |
| RUNNING | 收益中 |
| PAUSED | 收益暂停 |
| FINISHED | 收益结束 |

## 7.8 recharge_order.status

# 8. 核心流程映射

## 8.1 注册/登录流程

```text
用户输入邮箱
→ 检查 Redis 频率限制（email_code_rate:{email}:LOGIN，最多5次/分钟）
→ 发送验证码邮件，写 email_verify_code（只存哈希）
→ 用户输入验证码校验
→ 邮箱不存在 → 开启事务：
    创建 app_user
    创建 user_wallet
    写 user_referral_relation（有邀请码则关联上级）
    写 user_team_relation（从直属上级向上展开，写所有祖先关系）
    事务提交
→ 更新 email_verified_at、last_login_at
→ 生成 JWT Token 返回
```

## 8.2 推荐关系写入逻辑

假设新用户 X 使用 C 的邀请码注册，已有关系 A → B → C：

```text
user_referral_relation：
  X.parent_user_id = C
  X.level1_user_id = C
  X.level2_user_id = B
  X.level3_user_id = A

user_team_relation（闭包写入）：
  C → X，level_depth = 1
  B → X，level_depth = 2
  A → X，level_depth = 3
```

## 8.3 充值流程

```text
用户选择 recharge_channel
→ 校验：apply_amount >= MAX(sys_config.recharge.min_amount, channel.min_amount)
→ 用户填写金额、交易哈希、上传凭证
→ 提交 recharge_order（status=SUBMITTED）
→ 客服后台审核
→ 审核通过：
    user_wallet.available_balance += actual_amount（使用乐观锁）
    user_wallet.total_recharge += actual_amount
    写 wallet_transaction（tx_type=IN，biz_type=RECHARGE）
    recharge_order.status = APPROVED
    写 sys_notification（RECHARGE_SUCCESS）
```

## 8.4 下单支付机器费用流程

```text
用户选择 product / ai_model / rental_cycle_rule
→ 后端从数据库读取最新配置，计算 expected_daily_profit 和 expected_total_profit
→ 创建 rental_order（status=PENDING_PAY），写入所有快照字段
→ 用户确认支付
→ 校验 available_balance >= order_amount
→ user_wallet.available_balance -= order_amount（乐观锁）
→ 写 wallet_transaction（tx_type=OUT，biz_type=RENT_PAY）
→ rental_order.order_status = PAID
→ 后端生成 api_credential（token_status=GENERATED）
→ rental_order.order_status = PENDING_ACTIVATION
→ rental_order.api_generated_at = now()（超时1小时后自动取消）
```

## 8.5 API 激活（支付部署费）流程

```text
用户在控制台查看 API，点击"激活并支付部署费"
→ 校验 order_status = PENDING_ACTIVATION（其他状态拒绝）
→ 校验 available_balance >= deploy_fee_snapshot
→ 创建 api_deploy_order（status=PENDING_PAY）
→ user_wallet.available_balance -= deploy_fee_snapshot（乐观锁）
→ 写 wallet_transaction（tx_type=OUT，biz_type=API_DEPLOY_FEE）
→ api_deploy_order.status = PAID，paid_at = now()
→ api_credential.token_status = ACTIVATING，activation_paid_at = now()，activated_at = now()，auto_pause_at = now() + 24h
→ rental_order.order_status = ACTIVATING，activated_at = now()，auto_pause_at = now() + 24h
→ 写 sys_notification（API_ACTIVATED）
```

## 8.6 24小时后自动暂停流程（定时任务）

```text
任务名：auto_pause
扫描条件：order_status = ACTIVATING AND auto_pause_at <= now()
每条记录独立事务处理：
  rental_order.order_status = PAUSED，paused_at = now()
  api_credential.token_status = PAUSED，paused_at = now()
写 scheduler_log
```

## 8.7 激活超时自动取消流程（定时任务）

```text
任务名：activation_timeout_cancel
扫描条件：order_status = PENDING_ACTIVATION AND api_generated_at <= now() - sys_config.order.activation_timeout_minutes
每条记录独立事务处理：
  退还 order_amount 到 user_wallet.available_balance（乐观锁）
  写 wallet_transaction（tx_type=IN，biz_type=REFUND）
  api_credential.token_status = REVOKED
  rental_order.order_status = CANCELED，canceled_at = now()
  写 sys_notification（ORDER_CANCELED）
写 scheduler_log
```

## 8.8 用户启动收益流程

```text
用户点击"启动"
→ 校验 order_status = PAUSED
→ rental_order.order_status = RUNNING
→ rental_order.profit_status = RUNNING
→ rental_order.started_at = now()
→ rental_order.profit_start_at = now()
→ rental_order.profit_end_at = now() + cycle_days_snapshot天
→ api_credential.token_status = ACTIVE，started_at = now()
```

## 8.9 每日收益生成流程（定时任务）

```text
任务名：daily_profit
获取 Redis 分布式锁（scheduler:daily_profit:lock，TTL=3600s）
→ 获取失败则退出（上一次还在跑）
分页扫描（每页100条）：
  order_status = RUNNING
  AND profit_start_at <= now()
  AND DATE(profit_end_at) > today（UTC+8）
对每条订单：
  独立事务：
    INSERT IGNORE INTO rental_profit_record (rental_order_id, profit_date = today, ...)
    （唯一约束保证幂等，INSERT成功才继续，IGNORE则跳过）
    final_profit_amount = token_output_per_day_snapshot × token_unit_price_snapshot × yield_multiplier_snapshot
    user_wallet.available_balance += final_profit_amount（乐观锁）
    user_wallet.total_profit += final_profit_amount
    写 wallet_transaction（tx_type=IN，biz_type=RENT_PROFIT）
    rental_profit_record.status = SETTLED
  事务提交后触发佣金计算（独立事务）
释放 Redis 锁
写 scheduler_log
```

## 8.10 三级佣金生成流程

```text
收益入账成功后（独立于收益事务）：
查询 source_user 的 user_referral_relation
  level1_user_id → 读取 commission_rule（level=1，rate=0.20）→ commission_amount = profit × 0.20
  level2_user_id → commission_amount = profit × 0.10
  level3_user_id → commission_amount = profit × 0.05
对每个上级（独立事务）：
  INSERT IGNORE INTO commission_record (source_profit_id, level_no, ...)（唯一约束幂等）
  上级 user_wallet.available_balance += commission_amount（乐观锁）
  上级 user_wallet.total_commission += commission_amount
  写 wallet_transaction（tx_type=IN，biz_type=COMMISSION_PROFIT）
  commission_record.status = SETTLED
  写 sys_notification（COMMISSION_SUCCESS）
rental_profit_record.commission_generated = 1
```

## 8.11 到期结算流程（定时任务）

```text
任务名：order_expire_settle
扫描条件：order_status = RUNNING AND profit_end_at <= now()
每条记录独立事务：
  rental_order.order_status = EXPIRED，expired_at = now()
  rental_order.profit_status = FINISHED
  api_credential.token_status = EXPIRED，expired_at = now()
  生成 rental_settlement_order（settlement_type=EXPIRE）
  汇总累计收益（SUM rental_profit_record.final_profit_amount）填入 profit_amount（仅展示）
  actual_settle_amount = principal_amount（本金全额退还，或按业务规则）
  user_wallet.available_balance += actual_settle_amount（乐观锁）
  写 wallet_transaction（tx_type=IN，biz_type=SETTLEMENT）
  rental_settlement_order.status = SETTLED
  rental_order.settlement_status = SETTLED
  写 sys_notification（ORDER_EXPIRED）
写 scheduler_log
```

## 8.12 提前结算流程

```text
用户申请提前结算
→ 校验 order_status IN (ACTIVATING, PAUSED, RUNNING)
→ 生成 rental_settlement_order（settlement_type=EARLY_TERMINATE）
→ penalty_amount = order_amount × early_penalty_rate_snapshot
→ actual_settle_amount = order_amount - penalty_amount
→ 扣减违约金：写 wallet_transaction（tx_type=OUT，biz_type=EARLY_PENALTY）
→ 退还本金：user_wallet.available_balance += actual_settle_amount（乐观锁）
→ 写 wallet_transaction（tx_type=IN，biz_type=SETTLEMENT）
→ rental_order.order_status = EARLY_CLOSED
→ rental_order.profit_status = FINISHED
→ api_credential.token_status = REVOKED，revoked_at = now()
→ rental_settlement_order.status = SETTLED
```

## 8.13 提现流程

```text
用户提交提现申请：
→ 校验 apply_amount >= sys_config.withdraw.min_amount（10）
→ 校验今日累计（PENDING_REVIEW+APPROVED+PAID）+ apply_amount <= sys_config.withdraw.max_daily_amount（100000）
→ 校验提现地址格式（TRC20/ERC20/BEP20 正则，前后端双重校验）
→ 计算手续费：
    fee_amount = apply_amount >= fee_free_threshold(100) ? 0 : apply_amount × fee_rate(0.05)
    actual_amount = apply_amount - fee_amount
→ user_wallet.available_balance -= apply_amount（乐观锁）
→ user_wallet.frozen_balance += apply_amount
→ 生成 withdraw_order（status=PENDING_REVIEW）
→ 写 wallet_transaction（tx_type=FREEZE）

审核通过并打款：
→ user_wallet.frozen_balance -= apply_amount
→ user_wallet.total_withdraw += actual_amount
→ withdraw_order.status = PAID
→ 写 wallet_transaction（tx_type=OUT）
→ 写 sys_notification（WITHDRAW_SUCCESS）

审核驳回：
→ user_wallet.frozen_balance -= apply_amount
→ user_wallet.available_balance += apply_amount
→ withdraw_order.status = REJECTED
→ 写 wallet_transaction（tx_type=UNFREEZE）
→ 写 sys_notification（WITHDRAW_REJECTED）
```

## 8.14 封号流程

```text
后台管理员执行封号：
→ app_user.status = 0
→ 扫描该用户所有 order_status = RUNNING 的订单：
    rental_order.order_status = PAUSED，paused_at = now()
    api_credential.token_status = PAUSED，paused_at = now()
→ 写 sys_admin_log（action=BAN_USER）
已冻结提现单：保持原状，由后台人工处理
```

| 任务名 | 触发频率 | 说明 |

- 分布式锁防止多实例重复执行
- 唯一约束兜底，保证幂等

# 11. 重要开发口径

## 11.1 金额计算口径

```text
金额一律使用 BigDecimal。
数据库使用 DECIMAL(20,8)。
禁止用 double/float 计算金额。
手续费计算：BigDecimal.setScale(8, HALF_DOWN)。
```

## 11.2 时间计算口径

```text
所有时间使用 UTC+8（Asia/Shanghai）。
API 激活后 24 小时自动暂停：auto_pause_at = activated_at + 24小时。
用户点击启动后才设置 profit_start_at。
profit_end_at = profit_start_at + cycle_days_snapshot天。
每日收益按 profit_date（UTC+8日期）防重。
收益任务扫描：profit_date < DATE(profit_end_at)。
```

## 11.3 快照字段口径

下单时必须从数据库重新读取，不能信任前端传值：

```text
产品名称、GPU型号、显存、GPU算力、机器价格
每日Token产出
AI模型名称、AI厂商、Token单价、部署费用
周期天数、收益倍率、提前解约违约金比例
```

## 11.4 钱包写入口径

```text
任何余额变化必须写 wallet_transaction。
任何钱包扣款必须先校验 available_balance 是否足够。
所有余额变更必须使用乐观锁（version_no），更新行数为0则抛出异常。
提现必须走冻结余额流程。
每日收益和佣金入账必须幂等（唯一约束 + INSERT IGNORE）。
调账（ADJUST）仅 SUPER_ADMIN 可操作，必须写 sys_admin_log。
```

## 11.5 收益和佣金关系

```text
收益先入账给产生收益的用户（事务1）。
收益入账成功后，再触发三级佣金计算（独立事务，避免佣金失败导致收益回滚）。
佣金入账成功后，更新上级用户 total_commission。
一个收益记录最多生成三条佣金记录。
rental_profit_record.commission_generated = 1 标记已触发佣金。
```

## 11.6 提现地址校验口径

```text
前端按 network 选择对应正则校验。
后端接口层二次校验，不通过直接拒绝，不写数据库。
TRC20:     ^T[1-9A-HJ-NP-Za-km-z]{33}$
ERC20/BEP20: ^0x[0-9a-fA-F]{40}$
```

## 11.7 API Token 安全口径

```text
token_ciphertext 使用 AES-256-GCM 加密存储。
密钥从配置中心读取，禁止硬编码。
前台只展示 token_masked（脱敏），不返回明文。
```
